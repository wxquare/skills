# Code Review

将 Code Review 视为“在合并前拦截错误决策、隐藏耦合和生产风险”的过程，而不是格式检查。

先做高价值判断，再看局部实现。优先回答三件事：

- 这段代码会不会做错。
- 这段代码以后会不会很难改。
- 这段代码是不是把风险藏起来了。

## 工作流

### Step 1：建立评审边界

先确认你在审什么：

- 是单文件、函数、patch、PR，还是一个较大的功能分支。
- 这次改动解决什么问题，是否改变用户可见行为。
- 哪些文件是主改动，哪些只是伴随修改。

如果用户只给了一个模糊请求，如“帮我 review 一下”，先读取 diff、改动文件或目标代码，不要凭感觉给建议。

### Step 2：按从大到小的顺序审查

始终按下面顺序检查，不要一开始就陷入命名和格式：

1. 需求与行为：实现是否真的满足需求，而不是只覆盖 happy path。
2. 边界与设计：代码是否放在正确层次，是否引入跨层泄漏或额外耦合。
3. 正确性：状态迁移、并发、幂等、异常处理、空值、边界条件是否安全。
4. 可维护性：函数职责、抽象粒度、命名、重复逻辑、控制流复杂度是否合理。
5. 测试与运维：测试是否覆盖关键风险，日志、指标、灰度和回滚抓手是否充分。

如果在更高层已经发现结构性问题，优先报告该问题，不要被细枝末节分散注意力。

### Step 3：优先寻找高风险问题

重点盯以下几类风险：

- 状态一致性：是否可能部分成功、部分失败。
- 并发语义：是否会重复执行、重复扣减、脏写覆盖。
- 超时重试：重试是否幂等，是否会放大下游压力。
- 兼容性：接口、字段、枚举、状态值是否破坏旧调用方。
- 观测性：故障发生后是否能定位到哪一步出错。
- 回滚能力：出问题后是否有止损路径。

对于业务系统，这些问题通常比“代码风格不统一”更重要。

### Step 4：把发现写成可执行结论

输出 findings 时：

- findings 放在最前面。
- 按严重度排序，通常从高到低。
- 每条结论尽量包含：问题位置、具体症状、后果、建议方向。
- 引用文件和行号时保持精确。
- 如果没有发现明确问题，明确写“未发现实质性 findings”，并补充残余风险或测试盲区。

不要输出“代码整体不错，建议优化一下”这类低信息密度结论。

## 评审维度

结合本 skill 附带的 references，从以下维度组织判断：

- 编码边界：职责是否清楚，业务流程是否有显式编排层。
- 代码可读性：业务语言是否清晰，是否混入过多技术噪音。
- 变化隔离：变化点是否被隔离，抽象是否稳定。
- 错误语义：失败路径是否与成功路径一样被认真处理。
- 可测试性：是否容易隔离核心逻辑进行验证。
- Review 质量：是否优先看正确性、设计和风险，而非格式。

审查时优先读取：

- [references/review-checklist.md](references/review-checklist.md)：详细检查清单与严重度判断。
- [references/findings-template.md](references/findings-template.md)：输出格式、措辞模板与示例。

## 输出要求

默认输出结构：

1. Findings
2. Open Questions / Assumptions
3. Brief Summary

写 Findings 时遵循：

- 一条 finding 只讲一个核心问题。
- 用“问题 -> 影响 -> 建议”或“症状 -> 后果 -> 修复方向”组织。
- 如果证据不足，以“可能”“建议确认”表述，不要假装确定。
- 如果只是风格建议且不影响正确性、可维护性或风险控制，降低优先级或不写。

## 适用场景

适合：

- 审查 PR、diff、补丁或重构改动。
- 检查业务主链路实现是否偏离设计。
- 审查高风险系统改动，如下单、库存、支付、计价、审批、状态机。
- 判断一段代码是否 ready to merge。

不适合：

- 纯格式化检查或 lint 结果复述。
- 没有代码上下文时空泛讨论“怎么写更优雅”。
- 把完整设计工作推迟到 Code Review 阶段再重新发明方案。

## 审查准则

优先拦截这些情况：

- Handler、Use Case、Domain、Infrastructure 职责混杂。
- 长函数承载多种不相关业务责任。
- 异常被吞掉，错误语义无法向上层暴露。
- 重试、幂等、回滚、补偿语义不清楚。
- 新抽象只是“看起来高级”，但没有稳定收益。
- 提交过大，导致几乎无法认真审查。

如果发现问题本质上是架构或方案层问题，明确指出应上升到设计讨论，而不是只在局部做补丁式修复。
