# Code Review Checklist

按下面顺序审查，避免一开始就陷入局部实现。

## 1. 需求与行为

- 这次改动到底解决什么问题。
- 是否改变用户可见行为、接口契约或数据语义。
- 是否只实现了 happy path。
- 是否遗漏空输入、重复请求、无权限、数据不存在等边界情况。

## 2. 边界与设计

- 代码是否放在正确层次。
- 是否出现跨层依赖、隐藏耦合或职责泄漏。
- 关键业务流程是否有显式编排层。
- 新抽象是否真在隔离变化点，而不是增加理解成本。

## 3. 正确性与可靠性

- 是否可能出现部分成功、部分失败。
- 是否存在并发冲突、重入、重复执行或顺序错误。
- 重试是否幂等，是否会放大压力。
- 错误是否被吞掉，错误类型是否足以支持上层决策。
- 是否需要补偿、回滚、对账或状态修复。

## 4. 可维护性

- 命名是否表达业务语义。
- 函数是否过长，职责是否过多。
- 控制流是否过深，是否存在明显重复逻辑。
- 注释是否解释“为什么”，而不是复述代码。

## 5. 测试与可运维性

- 是否新增或更新了关键测试。
- 测试是否覆盖核心分支，而不是只覆盖表面路径。
- 是否有必要的日志、指标、trace 标签。
- 发布后是否有灰度、监控、回滚抓手。

## 严重度建议

- `P0`：会直接导致错误结果、资损、数据破坏或严重线上事故。
- `P1`：高概率正确性缺陷、明显设计错误、关键兼容性风险。
- `P2`：可维护性明显恶化、测试或观测性缺口、较高后续返工风险。
- `P3`：低风险改进项、表达或结构优化建议。

## 何时不写 finding

以下情况通常不值得单独写 finding：

- 纯个人风格偏好。
- 已被格式化工具自动解决的问题。
- 没有证据支持、只有模糊直觉的问题。
- 与本次改动无关的广泛重构欲望。
