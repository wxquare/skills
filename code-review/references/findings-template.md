# Findings Template

默认把 findings 放在回答最前面。

## 简洁模板

```text
1. [P1] <问题标题>
   <文件路径>:<行号>
   问题：<具体症状>
   影响：<为什么危险，可能造成什么后果>
   建议：<修复方向或建议确认的点>
```

## 叙述模板

```text
1. [P1] <问题标题> — <文件路径>:<行号>
   这里的实现会导致 <行为/风险>，因为 <原因>。在 <场景> 下可能出现 <后果>。建议 <修复方向>。
```

## 无 finding 时

```text
未发现实质性 findings。

残余风险 / 测试盲区：
- <如果有，补充仍建议关注的点>
```

## Open Questions / Assumptions

当证据不足但风险值得追问时，写在 findings 之后：

```text
Open questions / assumptions:
- 我假设 <前提>；如果不成立，上述结论需要重新评估。
- 建议确认 <接口语义 / 事务边界 / 幂等假设>。
```

## 示例

```text
1. [P1] 库存扣减成功后缺少失败补偿 — order/usecase.go:84
   这里先扣减库存，再写订单，但订单写入失败时没有补偿或回滚路径。
   在数据库短暂失败时会留下“库存已扣、订单不存在”的不一致状态。
   建议至少补充补偿逻辑，或显式引入事务边界 / saga 语义。
```

## 写作提醒

- 用证据说话，不用情绪判断。
- 指出具体风险，不要只说“这样不太好”。
- 如果只是建议而不是缺陷，降低严重度。
- 让作者看完后知道下一步该怎么改或该确认什么。
