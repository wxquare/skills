# TD — 技术方案设计文档写作与审查

一个 skill，两种模式：

- `/technical-design:write <输入>` — 将需求、笔记或会议纪要转化为结构完整的技术方案设计文档
- `/technical-design:review <输入>` — 审查已有 TD 文档的结构完整性、决策质量和风险覆盖度

框架源自《系统设计与架构实战》第 1 章的设计方法论（六维画布 + 8 段式骨架）。

---

## 模式一：technical-design:write — 写 TD

### 使用方式

```
/technical-design:write <需求描述、笔记或会议纪要>
/technical-design:write outline <输入>     # 仅生成提纲
/technical-design:write draft <输入>       # 生成完整草案
```

### 工作流程

#### Step 1：需求澄清与问题定义

先回答：
- 这个系统/功能服务谁？核心使用场景是什么？
- 当前方案或旧系统的痛点是什么？
- 哪些指标是必须达成的（延迟、吞吐、稳定性、正确性）？
- 哪些内容属于非目标，本期明确不做？
- 外部依赖有哪些？

#### Step 2：容量估算与真实约束

至少估算：平均/峰值 QPS、数据量、读写比、热点、RT 目标、SLA、外部约束。

#### Step 3：六维画布设计阶段（Design Phase）

在动手写 TD 之前，用六维画布把方案想清楚：

| 维度 | 核心问题 | 最少产出 |
|------|---------|---------|
| **Model** | 事实源是谁？状态怎么流转？实体边界在哪？ | 核心实体描述、状态机、事实源声明 |
| **Process** | 主链路怎么走？事务边界？失败了怎么补偿？ | 核心时序图、事务范围、异常处理策略 |
| **Capability** | 模块怎么划分？为什么选这个组件？ | 模块职责声明、选型决策记录 |
| **Governance** | 最坏情况？怎么兜底？对账和补偿怎么做？ | 风险清单、限流/熔断/降级策略、对账方案 |
| **Measurement** | 怎样算正常？出了事怎么发现？ | SLI/SLO、监控埋点、告警规则 |
| **Evolution** | 今天的设计会堵死明天的路？数据怎么迁移？ | 兼容策略、非目标假设、迁移路径草图 |

#### Step 4：按 8 段式骨架组织文档

```
1. Background / Objective / Scope
   为什么现在做、成功标准、本期不做什么

2. Current State / Problem Statement
   当前系统是什么样、痛点是否真实存在

3. Key Decisions（最关键的部分）
   最关键的 2-5 个设计决定、候选方案、为什么选这个、代价和 fallback

4. End-to-End Flow / Responsibility Split
   从输入到输出链路怎么走、谁在什么步骤负责什么、同步/异步边界

5. API / Data Contract
   字段含义、兼容性、幂等/顺序/版本/精度

6. Dependency / Rollout / Risk / Rollback
   依赖谁、blocker、上线顺序、降级和回退

7. Open Questions / Assumptions
   哪些没定、基于什么假设推进、哪些结论等外部条件

8. Work Breakdown / Owner / Timeline
   工作拆分、负责人、先后依赖、设计完成标准
```

#### Step 5：关键决策写法

每条决策必须包含五项：

- **候选方案**：至少 2 个可行方案
- **选择结果**：明确选哪个
- **选择依据**：对比分析，不说"方案 B 更好"，说"因为……所以……"
- **明确代价**：选这个放弃/牺牲了什么
- **Fallback**：选错了怎么退，后续迁移路径

推荐写法：

```text
决策：<最终选择>

候选方案：
- 方案 A：<简述>
- 方案 B：<简述>

选择依据：
- <为什么选 B 而不是 A>

代价：
- <选 B 带来的具体成本>

Fallback：
- <如果选错了，怎么退>
```

### 输出产物

- **outline**：8 段式骨架 + 每段要点提示
- **draft**：完整的 8 段内容，包含关键决策分析、核心链路描述、风险清单、灰度/回滚方案、开放问题和任务拆分建议

### 注意事项（write 模式）

- **先想再写**：不要直接跳到接口和表结构
- **先讲决策，再讲细节**：关键决策是评审者最关心的
- **区分设计与实现**：TD 说明系统行为，不是实施手册
- **异常路径和正常路径同等重要**：超时、重试、幂等、补偿、降级都需要设计

## 模式二：technical-design:review — 审 TD

### 使用方式

```
/technical-design:review <方案文档内容或文件路径>
/technical-design:review quick <方案文档>     # 快速检查（仅 A + E 维度）
/technical-design:review deep <方案文档>      # 深度审查（全量 5 维）
```

### 审查维度

#### Dimension A：骨架完整性（8 段式检查）

检查每段是否覆盖了核心问题：

| 段落 | 核心问题 | 通过标准 |
|------|---------|---------|
| 1. Background / Objective / Scope | 为什么现在做？成功标准？不做什么？ | 有明确背景、目标和边界声明 |
| 2. Current State / Problem Statement | 当前痛点？是否真实可验证？ | 有现状描述，痛点可验证 |
| 3. Key Decisions | 关键决策？有选项和分析？ | ≥ 2 个决策，每项有选项+依据+代价+fallback |
| 4. End-to-End Flow | 核心链路？谁负责什么？ | 流程图/时序图，有同步/异步边界 |
| 5. API / Data Contract | 字段含义？幂等/版本？ | 关键字段有语义说明，幂等键有定义 |
| 6. Dependency / Rollout / Risk / Rollback | 依赖？上线？回退？ | 依赖集中列出，灰度有维度，回滚可执行 |
| 7. Open Questions / Assumptions | 哪些没定？基于什么假设？ | 开放问题有显式出口，假设有验证条件 |
| 8. Work Breakdown / Owner / Timeline | 工作拆分？Owner？ | 有任务拆分、Owner 和先后依赖 |

严重度：🔴 缺失 → 🟡 过浅 → 🟢 可优化 → ✅ 完整

#### Dimension B：六维设计画布覆盖度

| 维度 | 核心问题 |
|------|---------|
| **Model** | 事实源？状态机？实体边界？ |
| **Process** | 主链路？事务边界？补偿策略？ |
| **Capability** | 模块划分？选型依据？ |
| **Governance** | 风险清单？限流/熔断？对账方案？ |
| **Measurement** | SLI/SLO？监控埋点？告警规则？ |
| **Evolution** | 兼容策略？迁移路径？ |

#### Dimension C：决策质量评估

每条决策检查五项：✅ 候选方案（≥2） ✅ 选择结果 ✅ 选择依据 ✅ 明确代价 ✅ Fallback

缺失 ≥3 → 🔴 | 2 → 🟡 | 1 → 🟢 | 全部 → ✅

#### Dimension D：风险/灰度/回滚可执行性

- **风险**：是什么、何时发生、影响范围、如何发现、如何缓解、缓解失败怎么回滚
- **灰度**：一次性/灰度？灰度维度？观测指标？失败阈值？谁拍板？
- **回滚**：回滚对象？谁执行？回滚前检查什么？如何确认恢复？

任何项模板占位 → 🔴 | 缺 1-2 项细节 → 🟡 | 全部可执行 → ✅

#### Dimension E：反模式检测

| 反模式 | 判断方法 | 严重度 |
|--------|---------|--------|
| 方案写成实施手册 | 命令/SQL/接口细节多，缺少主设计决策 | 🟡 |
| 选项列了但没有决策标准 | A/B 并列展示，没有为什么选这个 | 🟡 |
| 字段语义不统一 | 金额/状态在不同章节含义漂移 | 🔴 |
| 依赖和 blocker 散落正文 | 读者需要自己拼图 | 🟡 |
| rollout/rollback 模板占位 | 看起来写了，实际上不可执行 | 🔴 |
| 开放问题没有显式出口 | 未定内容埋在自然语言里 | 🟡 |
| 设计与运行手册混写 | 评审想看系统行为，却被操作步骤淹没 | 🟢 |

### 输出格式

快速检查（quick）：

```
## TD Review 报告（快速检查）

**结论**：[通过 / 有条件通过 / 不通过]

### 关键缺失
- [{维度}] — 描述

### 主要反模式
- [{反模式}] — 描述
```

深度审查（deep）：

```
## TD Review 报告（深度审查）

### A. 骨架完整性
| 段落 | 状态 | 说明 |

### B. 六维画布覆盖度
| 维度 | 状态 | 说明 |

### C. 决策质量
| 决策 | 选项 | 依据 | 代价 | Fallback | 结论 |

### D. 风险/灰度/回滚
- 风险：[可执行 / 模板占位]
- 灰度：[可执行 / 模板占位]
- 回滚：[可执行 / 模板占位]

### E. 反模式检测
- [{反模式}] — 描述

### 改进优先级
1. [最高优先] 🔴
2. [次优先] 🟡
3. [可选] 🟢
```

### 注意事项（review 模式）

- **不审查代码正确性**：只评估方案文档的结构完整性和设计覆盖度
- **不替代人评审**：评审辅助工具，不替代架构师的专业判断
- **优先识别结构性缺失**：段落缺失、决策不完整、反模式 > 内容深度问题

## 关于上下文

- 如果用户只给了模糊的需求描述，先帮用户澄清需求，再进入写作流程
- 如果用户给的是文档内容要求审查，直接按 review 流程处理
- 如果用户同时提供了需求描述和期望的文档格式，按 write 流程生成
